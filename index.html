Java-线程

发表于 2021-01-06 更新于 2021-01-09 分类于 Java 阅读次数：
本文字数： 4.2k 阅读时长 ≈ 4 分钟

简要描述

  • 进程与线程
    • 进程：是操作系统进行资源分配的最小单位，是程序在计算机上的一次执行活动，进程内的线程
    • 线程：是CPU调度的最小单位，必须寄生于进程存在
  • 并发与并行
    • 并发：单位时间内，执行的任务量，如单位时间内CPU执行了多少次任务
    • 并行：同时执行的任务，比如一个人边吃饭边看电视，如多线程的模式

线程的创建方式

  • new Thread方式
  • implements Runnable方式

    扩展{Callable，Future}

    前面两种方式存在一个明显的缺陷就是当线程执行完毕后无法获取返回值
    因此从Java1.5开始提供了Callable,Future来解决这一问题

备注：Callable并不属于线程的另一种创建方式，它只是对Runnable的一种封装，具体可见Thread源码注释和下面的的使用分析

线程的生命周期共有六种状态，可在源码Thread#State内部类中进行查看

常用方法

  • Thread.start()
    线程进入就绪队列等待分配CPU，待分配到CPU后就会执行run()方法，start()不能重复调用

  • Thread.yield()
    让出CPU使用权，给其他线程执行机会，RUNNING -> READY

  • Thread.join()
    在当前线程中插入其他线程执行过程，插入线程执行完毕后当前线程继续执行t2.join()后的代码块
    插入线程执行过程中当前线程处于等待状态(WAITING/TIMED_WAITING)，join可设置最长等待执行时间

  • Thread.sleep()/sleep(long millis)
    用来暂停当前线程的执行，通知线程调度器把当前线程在指定的时间内置为WAITING状态，当wait时间到达，线程状态重新变为READY并等待CPU再次调度执行

  • Thread.interrupt()
    将线程的中断状态置为true，只是通知线程可以中断，并不会强行中断正在执行的线程
    如果线程处于阻塞状态interrupt()方法被调用时，中断状态会重置为false，并抛出InterruptedException异常

  • Object.wait()/wait(long millis)
    将线程放入等待池(Waiting Pool)，同时放弃对象锁，wait后返还对象锁
    线程必须拥有当前对象的锁，因此wait()需在synchronized块中调用，则否会抛出IllegalMonitorStateException异常

  • Object.notify()/notifyAll()
    唤醒当前对象等待池中等待的第一个/所有线程
    notify()/notifyAll()也必须拥有相同对象锁，需在synchronized块中调用，否则也会抛出IllegalMonitorStateException异常

Handler机制是Android系统中非常核心的线程间通讯机制，广泛的应用在系统通知中，包括常见的Activity生命周期的通知，手势通知等等
Handler机制Java层涉及主要的四个类包括Handler，Looper，MessageQueue，Message；该机制构成了典型的生产者和消费者模型
接下来将对Java层的运行原理进行分析

机制解读

  • 1.Handler负责发送和接收消息
  • 2.Looper负责不断的轮询MessageQueue中存在的消息，其中使用了ThreadLocal保证了一个线程只有一个Looper，具体实现可查看Looper.prepare()
  • 3.MessageQueue存储Message的容器
  • 4.主线程的Looper是在ActivityThread.mian()进行启动，主线程的Looper不允许退出

扩展点

  • HandlerThread

    • 本质：继承Thread，封装Looper；内部维护了Looper，run()方法中首先调用Looper.prepare()和Looper.loop()，quit()调用了Looper.quit()

      相关问题

  • 1.一个线程有几个Handler?
    一个线程可以有多个Handler，但底层对应同一个Looper和同一个MessageQueue

  • 2.一个线程有几个Looper?如何保证？
    一个线程只有一个Looper，使用通过ThreadLocal来保证唯一性的

  • 3.Handler内存泄漏原因？为什么其他的内部类没有这个问题？
    Handler和Activity生命周期不一致，主线程中Looper(MessageQueue)的生命周期为应用的生命周期；当Activity中发送延时消息立即退出页面，此时延时Message可能还未执行，还持有Handler(Looper->MessageQueue->Message-Handler)引用，因此造成内存泄漏
    其他内部类没有这个问题是因为没有延时消息的机制

  • 4.为何主线程可以new Handler？如果想在子线程中new Handler要做什么准备？
    主线程的Looper.prepare()在ActivityThread.main()方法中已经调用了，因此可以直接new Handler
    子线程中new Handler之前需要准备子线程Looper，即Looper.prepare()，Looper.loop()，Handler.getLooper().quit()方法的调用

  • 5.子线程中维护的Looper，消息队列无消息的时候的处理方案是什么？有什么用？
    Handler.getLooper().quitSafely()，结束Looper轮询，释放资源

  • 6.既然可以存在多个Handler往MessageQueue中添加数据，它的内部是如何确保线程安全的？
    MessageQueue.enqueueMessage()的方法主体使用了synchronized关键字保证线程安全性

  • 7.我们使用Message时应该如何创建？
    Message.obtain()
    Message.recycle(),Message.recycleUnchecked()销毁机制并没有重复的创建对象，而只是将Message的内容擦除并重复使用

  • 8.Looper死循环为什么不会导致应用卡死？
    MessageQueue没有消息时，程序会阻塞在Looper的queue.next()中的nativePollOnce()，此时主线程会释放CPU资源进入休眠状态，直到下一次添加新消息时通知底层唤醒使其继续工作(MessageQueue.enqueueMessage() -> nativeWake(mPtr))
    此处使用Linux系统的epoll-I/O事件通知机制

本文字数： 1.1k 阅读时长 ≈ 1 分钟
简要描述
  • Java 注解（Annotation）又称 Java 标注，是 JDK5.0 引入的一种注释机制。 注解是元数据的一种形式，提供有关
    于程序但不属于程序本身的数据。注解对它们注解的代码的操作没有直接影响
自定义
元注解

  • 1.@Documented

    用于Java DOC工具提取成文档

  • 2.@Inherited

    表示允许子类继承父类中定义的注解

  • 3.@Target

    注解标记另一个注解，以限制可以应用注解的 Java 元素类型。目标注解指定以下元素类型之一作为其值：
    ElementType.ANNOTATION_TYPE 可以应用于注解类型
    ElementType.CONSTRUCTOR 可以应用于构造函数
    ElementType.FIELD 可以应用于字段或属性
    ElementType.LOCAL_VARIABLE 可以应用于局部变量
    ElementType.METHOD 可以应用于方法级注解
    ElementType.PACKAGE 可以应用于包声明
    ElementType.PARAMETER 可以应用于方法的参数
    ElementType.TYPE 可以应用于类的任何元素

  • 4.@Retention

    注解指定标记注解的存储方式：
    RetentionPolicy.SOURCE - 标记的注解仅保留在源级别中，并被编译器忽略
    RetentionPolicy.CLASS - 标记的注解在编译时由编译器保留，但 Java 虚拟机(JVM)会忽略
    RetentionPolicy.RUNTIME - 标记的注解由 JVM 保留，因此运行时环境可以使用它

  • 5.其他

    @SuppressWarnings
    …

在通用代码中，称为通配符的问号（ ? ）表示未知类型。通配符可以在多种情况下使用：作为参数，字段或局部变量的类型；有时作为返回类型（尽管更具体的做法是更好的编程习惯）

泛型擦除

Java语言引入了泛型，以在编译时提供更严格的类型检查并支持泛型编程。 为了实现泛型，Java编译器将类型擦除应用于：

  • 如果类型参数不受限制，则将通用类型中的所有类型参数替换为其边界（上下限）或 Object 。因此，产生的字节码仅包含普通的类，接口和方法
  • 必要时插入类型转换，以保持类型安全
  • 生成桥接方法以在扩展的泛型类型中保留多态
  • 类型擦除可确保不会为参数化类型创建新的类；因此，泛型不会产生运行时开销
